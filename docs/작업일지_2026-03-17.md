### 9.2 회원가입 플로우

#### 전체 흐름

```
[초대코드 입력] → [Google OAuth 로그인] → [역할 선택] → [기관 배정] → [관리자 승인 대기] → [서비스 이용]
```

#### 상세 단계

1. **초대코드 입력** (`/auth/invite`)
   - 관리자가 기관별 초대코드 발급 (예: `CAREVIA-SCHOOL-A1B2`)
   - `invitation_codes` 테이블에서 유효성·사용 횟수 검증
   - 초대코드에 기관 ID가 연결되어 자동 기관 배정

2. **Google OAuth 로그인** (`/login` → Supabase Auth)
   - `supabase.auth.signInWithOAuth({ provider: 'google' })`
   - 콜백: `/auth/callback` → Supabase 세션 확립
   - `handle_new_user` 트리거로 `profiles` 자동 생성 (현재 스키마에 이미 존재)

3. **역할 선택** (`/auth/setup`)
   - 신규 사용자(`profiles.role`이 기본값 'teacher')에게 역할 선택 UI 표시
   - teacher / parent 선택 → `profiles.role` 업데이트
   - parent 선택 시 자녀 연결 (`students.parent_id`) 추가 단계

4. **관리자 승인**
   - `profiles` 테이블에 `status` 컬럼 추가 (`pending` / `approved` / `rejected`)
   - 미승인 사용자는 "승인 대기 중" 화면만 표시
   - 관리자 대시보드에서 승인/거절 처리

#### AuthContext 변경 사항

```typescript
// AS-IS: localStorage 기반
const stored = loadAuth(); // localStorage에서 읽기

// TO-BE: Supabase 세션 기반
const { data: { session } } = await supabase.auth.getSession();
const { data: profile } = await supabase
  .from('profiles')
  .select('*')
  .eq('id', session.user.id)
  .single();

// onAuthStateChange 리스너로 세션 변경 감지
supabase.auth.onAuthStateChange((event, session) => { ... });
```

### 9.3 역할/권한 관리

#### 3역할 체계

| 역할 | 접근 범위 | 주요 기능 |
|------|-----------|-----------|
| **teacher** | 소속 기관 학생 전체 | 대시보드, 일일보고, 소통방, 행동분석, 케어팀 |
| **parent** | 자신의 자녀만 | 자녀 대시보드, 등원 전 한마디, 소통방, 행동 통계 |
| **admin** | 전체 기관/사용자 | 사용자 관리, 승인, 기관 설정, 전체 데이터 조회 |

#### RLS 정책 현황 및 보완

현재 `schema.sql`에 기본 RLS 정책이 구현되어 있음:
- `get_my_role()`, `get_my_org()` 헬퍼 함수 ✅
- 기관별 데이터 격리 (`organization_id` 기반) ✅
- 보호자 자녀 접근 (`parent_id = auth.uid()`) ✅

**추가 필요 정책:**
- `morning_reports`: 보호자 본인만 작성, 교사/admin 조회 가능
- `care_team`: 같은 기관 교사 조회, admin 전체 관리
- `invitation_codes`: admin만 CRUD, 일반 사용자는 검증용 SELECT만
- `profiles.status` 기반 접근 제한: `approved` 상태만 데이터 접근 허용

#### 관리자 승인 워크플로우

```
[신규 가입] → profiles.status = 'pending'
     ↓
[관리자 대시보드] → 가입 요청 목록 조회 (admin-v2/PendingApprovals)
     ↓
[승인] → profiles.status = 'approved' → 사용자에게 알림
[거절] → profiles.status = 'rejected' → 사유 안내
```

### 9.6 보안 고려사항

#### Row Level Security (RLS)

- **현재**: 7개 테이블 RLS 활성화 + 기본 정책 구현 완료 ✅
- **추가**: 신규 5개 테이블 RLS 정책 작성 필요
- **강화**: `profiles.status = 'approved'` 조건을 모든 데이터 테이블 정책에 추가
  ```sql
  -- 예: 승인된 사용자만 students 조회 가능
  create policy "students_select_approved" on public.students
    for select using (
      exists (select 1 from profiles where id = auth.uid() and status = 'approved')
      and (organization_id = get_my_org() or ...)
    );
  ```

#### 환경변수 관리

| 변수 | 용도 | 노출 위험 |
|------|------|-----------|
| `VITE_SUPABASE_URL` | Supabase 프로젝트 URL | 낮음 (공개 가능) |
| `VITE_SUPABASE_ANON_KEY` | Supabase anonymous key | 낮음 (RLS로 보호) |
| Google OAuth Client Secret | OAuth 인증 | **Supabase 대시보드에만 저장** (코드에 포함 금지) |

- `VITE_` 접두사 변수는 클라이언트에 번들링됨 → 민감 정보 포함 금지
- 서버 측 시크릿은 Supabase Dashboard 또는 Railway 환경변수에만 저장

#### CORS 설정

- Supabase 프로젝트 설정에서 허용 도메인 지정:
  - `http://localhost:5173` (개발)
  - `https://{railway-app}.up.railway.app` (프로덕션)

#### Rate Limiting

- Supabase 기본 rate limit 활용 (무료: 500 req/sec)
- 클라이언트 측 디바운싱: 소통방 메시지 전송, 검색 등
- 보호자 등원 전 한마디: 하루 1회 제한 (DB unique constraint로 보장)
